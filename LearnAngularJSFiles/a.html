                                                                                不是的，DOM操作部分应该放在哪里，要看具体的情况。当然要说放在 link 中一定是可行的，我想也应该是正确的吧。

                                                                                对于这个问题，当初我的思路是，首先，搞明白了指令的解析与执行的流程，文章中有提到。其次是，ng 的文档中有描述，为什么要分成 compile 与 link 这两个部分，这与其所谓的“数据双向绑定”有关，即 ng 模板的一个思路：

                                                                                http://code.angularjs.org/1.0.... #Understanding View

                                                                                >> Angular is different. The Angular compiler consumes the DOM with directives, not string templates. The result is a linking function, which when combined with a scope model results in a live view.

                                                                                至此，我觉得问题就很明白了。如果你的指令只是修改一下 DOM ，并且这个修改是一次性的，那么在 compile 阶段完成这个事就好了，根本不需要后面的 link 阶段。

                                                                                如果你的指令是生成一个与数据有关的 DOM 结构，那显然你需要在 link 阶段去完成数据的绑定操作，这时如果 DOM 结构是根据数据来的，那么只能在 link 这里做修改 DOM 的事了。

                                                                                总的来说，我觉得不应该从要做什么事这个角度去考虑一个指令（我要修改DOM，我要绑定数据之类），而应该是另外一个维度，从流程化的思维来考虑一 个指令（我先要生成一个架子，使用div把一些东西包起来。然后去解析各种来源的数据，然后把通过这些数据生成的一些节点放到我的架子中去。当这些数据变 化时，我又要做什么）。这样想的话，compile 和 link 中都会涉及修改 DOM 的操作，但是它们背后承载的业务概念是不同的。
